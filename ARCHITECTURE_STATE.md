## 2. Recovery Summary (2026-05-08)

### Incident
The API container was stuck in a restart loop. Every start attempt aborted during Drizzle ORM migration execution before the HTTP server could bind.

### Root Cause
`0007_remarkable_ben_urich.sql` — auto-generated by `drizzle-kit` — contained:

```sql
ALTER TYPE "public"."subscription_status" ADD VALUE 'expired';
```

Because `0004a_add_expired_enum_value.sql` was written manually (no matching snapshot in `meta/`), `drizzle-kit` did not know `'expired'` had already been added when it generated `0007` from the snapshot diff `0003_snapshot → 0007_snapshot`. By the time `0007` ran inside Drizzle's outer transaction, `ensureEnumValues()` had already committed `'expired'` in autocommit mode → PostgreSQL error `42710 duplicate_object`.

### Fix
Wrapped the problematic ALTER TYPE in `0007` with an idempotent `DO $$` block:

```sql
DO $$
BEGIN
    ALTER TYPE "public"."subscription_status" ADD VALUE IF NOT EXISTS 'expired';
EXCEPTION
    WHEN duplicate_object THEN NULL;
END;
$$;--> statement-breakpoint
```

### False Leads
- Suspected `0004a` was the source (it was not — different SQL format, no schema prefix)
- Suspected stale Docker image layer (layer caching is not the issue; `--no-cache` was used)
- Suspected volume bind mount overriding files (only `/app/uploads` is mounted)

### Result
All 4 pending migrations (`0004a`, `0004b`, `0004c`, `0007`) applied in sequence. API started successfully.

## 3. Migration Governance Rules

> See also: `docs/deployment/MIGRATION_GOVERNANCE.md`

1. **Never write raw `ALTER TYPE ... ADD VALUE` in production migrations.** Always use `IF NOT EXISTS` or a `DO $$ EXCEPTION` guard.
2. **All DDL migrations must be idempotent.** Assume any migration may be retried after a partial failure.
3. **Manual migrations must be reconciled against future drizzle-kit output.** If you write a migration by hand and it changes the schema, run `drizzle-kit generate` immediately afterward and inspect the output to ensure drizzle-kit's snapshot is updated.
4. **Never trust generated migrations blindly after manual SQL.** Read every generated `.sql` file before deploying.
5. **Inspect snapshot diffs** in `lib/db/drizzle/meta/` before every deploy to catch duplicate DDL.
6. **Never manually edit `drizzle.__drizzle_migrations`** unless production recovery requires it and a database backup has been taken and verified.
7. **Always take a database backup after successful recovery.**
8. **`ensureEnumValues()` in `migrate.ts`** must be kept in sync with any enum value additions across all migration files.

## 4. Multi-Tenant Boundary Model

> See also: `docs/architecture/MULTI_TENANT_BOUNDARIES.md`

| Boundary | Purpose | Isolation level |
|---|---|---|
| **Organization** | Security boundary | Hard — enforced server-side on every query |
| **Project** | Workspace / business grouping | Soft — filtering only, never security |

### Rules
- All tenant isolation **must** be enforced by `organization_id`.
- Project-level filtering must never substitute for organization-level enforcement.
- RLS policies are initialized on startup and scope to organization ownership.
- Cross-organization data access is forbidden by default.
- Server-side enforcement is authoritative. UI hiding alone is never sufficient.

| Role | Capabilities |
|---|---|
| System owner | Full system access, org management |
| Admin | Org-wide admin, user management |
| Project manager | Manage projects, assign roles |
| Document controller | Upload, transmit, issue documents |
| Reviewer | Review and comment |
| Member | Standard document access |
| Viewer | Read-only within assigned scope |
| Read-only override | Explicit read-only regardless of role |

### Restrictions
- **Trial / expired organizations**: module access restricted; upload limits enforced.
- **Module-based access**: each module (Document, NOC, AI, etc.) has its own entitlement check.
- Server-side middleware enforces all restrictions. Frontend UI restrictions are UX only.
