## المشاكل والأخطاء

**الخطأ الأكثر إيلاماً — جدول مفقود في Production:**
```
Error: relation "table_name" does not exist
```
السيناريو: migration جديدة أضافت جدولاً محلياً. في الـ production، المهاجرة لم تُشغَّل لسبب ما. الـ API crashes عند أول محاولة وصول للجدول.

التعليق الصادق: هذا الخطأ توقف بسببه deployment كامل. الحل كان يدوياً: تشغيل المهاجرة يدوياً على الـ production database ثم إعادة تشغيل الـ container.

**مشكلة Enum values في PostgreSQL:**
PostgreSQL لا يسمح بإضافة enum value جديدة ثم استخدامها في نفس الـ transaction. يجب أولاً commit الـ transaction ثم استخدام القيمة في transaction جديدة.

```sql
-- هذا يفشل:
BEGIN;
ALTER TYPE status_enum ADD VALUE 'new_value';
UPDATE table SET status = 'new_value' WHERE ...;  -- خطأ!
COMMIT;

-- الصواب: migration منفصلتان
-- Migration 1: ALTER TYPE ... ADD VALUE
-- Migration 2: UPDATE table ...
```

**مشكلة connection string مع underscores:**
URL parser كان يفشل مع database names تحتوي على underscores (مثل `edms_db`). الحل: استخدام URL parsing library مختلفة.

## القرارات المعمارية

**PostgreSQL = Internal only:**
قرار أمني حاسم: قاعدة البيانات لا تستمع على أي port من الخارج. الـ API فقط يتحدث معها عبر Docker internal network. هذا يعني أنه حتى لو كان الـ firewall معطلاً، قاعدة البيانات غير قابلة للوصول من الإنترنت.

**"free" → "expired" — تغيير semantic:**
كلمات المنتج مهمة. "Free plan" تجعل المستخدم يعتقد أنه يحصل على شيء مجاناً. "Expired" تخبره بوضوح: فترتك انتهت. هذا يقلل من confusion ويُوضح الموقف.

## الدروس المستفادة

1. **كل migration يجب أن تكون idempotent** — "CREATE TABLE IF NOT EXISTS" وليس "CREATE TABLE"
2. **PostgreSQL enum quirk حقيقية** — تحتاج لـ 2 migrations منفصلتين
3. **Database = internal network فقط** — هذا يجب أن يكون الـ default، وليس استثناء
4. **اسم الـ plan يؤثر على السلوك المتوقع** — naming is design
5. **Deployment يُكسر بأبسط الأخطاء** — جدول مفقود = 100% downtime
