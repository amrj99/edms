## ما تم بناؤه

**Trial Mode للمنظمات الجديدة:**
- أي منظمة جديدة تبدأ بفترة تجريبية تلقائية
- التحقق من البريد الإلكتروني مطلوب لإكمال التسجيل
- نهاية الـ trial → النظام يحول المنظمة لـ expired plan تلقائياً

**تحسينات الأمان:**
- تسجيل محاولات تسجيل الدخول الفاشلة
- سياسة lockout: حساب يُقفل بعد عدد من المحاولات الفاشلة
- تحسين رسائل الخطأ لتكون مفيدة للمستخدم بدون كشف معلومات حساسة

**Cloudflare R2 Integration:**
- الملفات المرفوعة تذهب الآن إلى Cloudflare R2 (Object storage)
- بدلاً من: local disk على الـ VPS
- R2 مثل S3 لكن بدون رسوم egress
- تحديث الكود لدعم R2 SDK مع authentication

**تقييد رؤية المشاريع:**
- المستخدم العادي لا يرى مشاريع المنظمات الأخرى في التقارير والقوائم
- قبلاً: بعض الـ endpoints كانت تُعيد بيانات عبر المنظمات

## المشاكل والأخطاء

**مشكلة R2 Setup:**
R2 يستخدم نفس API كـ S3، لكن:
- الـ endpoint مختلف (Cloudflare-specific)
- الـ authentication يستخدم R2_ACCESS_KEY و R2_SECRET_KEY (وليس AWS credentials)
- بعض الـ SDK calls كانت تفترض behaviors خاصة بـ AWS S3

**مشكلة Trial expiry:**
- كيف تحول منظمة من trial إلى expired بشكل تلقائي؟
- الحل: scheduled job يعمل دورياً (cron-style) ويتحقق من تواريخ الانتهاء
- التحدي: ماذا يحدث للبيانات عند انتهاء الـ trial؟ قرار: البيانات تبقى، الوصول يُقيَّد فقط

**مشكلة Login lockout UX:**
- الرسالة الأولى كانت: "User is locked" — غير مفيدة
- التحسين: "حسابك مؤقتاً معطل. انتظر X دقيقة أو تواصل مع المسؤول"

## القرارات المعمارية

**Cloudflare R2 بدلاً من local storage أو AWS S3:**

| الخيار | المزايا | العيوب |
|--------|--------|--------|
| Local disk (VPS) | بسيط، سريع | لا يتوسع، فقدان عند failure |
| AWS S3 | معيار صناعي | رسوم egress مرتفعة |
| Cloudflare R2 | لا egress fees، قريب من Workers | أحدث، documentation أقل |

الاختيار: R2 — المشروع يستخدم Cloudflare Workers AI أصلاً، فالتكامل طبيعي.

**Trial → Expired (بدون حذف البيانات):**
قرار: البيانات لا تُحذف أبداً عند انتهاء الـ trial. فقط الوصول يُقيَّد. السبب: فقدان بيانات العملاء حتى في الـ trial يُضر بالثقة ويسبب مشاكل قانونية محتملة.

## الدروس المستفادة

1. **Object storage أفضل من local disk في كل حال** — حتى في المشاريع الصغيرة، disk failure = فقدان ملفات
2. **R2 = S3 API لكن ليس S3 بالضبط** — افحص التوثيق بدقة خاصة للـ multipart upload
3. **Login lockout = توازن بين أمان وـ UX** — lockout قصير جداً → مزعج، طويل جداً → خطر
4. **Trial periods تحتاج state machine واضحة** — trial → active → expired → cancelled، وكل حالة لها سلوك مختلف

## مصطلحات جديدة

| المصطلح | الشرح |
|---------|-------|
| **Object Storage** | تخزين الملفات كـ "objects" في cloud بدلاً من filesystem تقليدي (مثل S3, R2) |
| **Egress fees** | رسوم نقل البيانات من السحابة للخارج — R2 لا تحتسبها بعكس AWS |
| **Lockout policy** | سياسة تقييد الوصول بعد محاولات فاشلة متعددة |
| **Cron job** | مهمة تعمل تلقائياً بجدول زمني (كل ساعة، كل يوم، إلخ) |
| **Idempotent** | عملية تعطي نفس النتيجة مهما كررتها — مهمة في الـ migrations والـ cron jobs |
