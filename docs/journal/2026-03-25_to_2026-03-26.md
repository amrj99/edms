## المشاكل والأخطاء

**التحدي الأكبر في هذه الفترة:** بناء كل شيء دفعة واحدة — خطر حقيقي.
- عندما تبني نظاماً كاملاً في يومين، يصعب التتبع الدقيق لكل قرار
- بعض الـ endpoints بُنيت بدون تفكير كافٍ في الـ authorization
- هذا ظهر لاحقاً في أبريل كـ security debt

**مشكلة معمارية اكتُشفت لاحقاً:**
- الـ `canAccessProject` function لم تكن تتحقق من الـ org_id بشكل صحيح
- أي مستخدم يعرف project ID يستطيع الوصول لمستنداته — ثغرة cross-org

## القرارات المعمارية

| القرار | البديل الذي رُفض | السبب |
|--------|----------------|-------|
| pnpm monorepo | مشاريع منفصلة | مشاركة الكود والـ dependencies |
| Express + Node.js | Fastify / NestJS | سرعة البدء والمرونة |
| PostgreSQL | MongoDB | الـ EDMS تعمل على بيانات علائقية (documents، revisions، transmittals) |
| JWT + sessions | OAuth only | التحكم الكامل في المصادقة |
| React + Vite | Next.js | هذا SPA خالص، لا حاجة لـ SSR |

## الدروس المستفادة

1. **السرعة الأولية مكلفة لاحقاً** — الكود المكتوب بسرعة يحتاج re-hardening بعدها
2. **Authorization ليس ميزة، هو بنية تحتية** — يجب التفكير فيه من اليوم الأول
3. **Multi-tenancy معقد** — عزل البيانات بين organizations أكثر تعقيداً مما يبدو

## مصطلحات جديدة

| المصطلح | الشرح |
|---------|-------|
| **Monorepo** | مشروع واحد يحتوي على عدة packages أو تطبيقات تشترك في الـ dependencies |
| **pnpm** | مدير حزم أسرع من npm مع workspace support |
| **JWT** | JSON Web Token — طريقة للمصادقة بدون sessions في الـ database |
| **Multi-tenancy** | قدرة النظام على خدمة عدة عملاء (منظمات) من نفس الكود والـ database |
| **Transmittal** | في EDMS: حزمة رسمية من المستندات تُرسل لطرف آخر مع رقم مرجعي |
