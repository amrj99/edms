# كيفية تحديث الأرشيف بطريقة اقتصادية

الهدف: إنشاء قيد جديد لكل 7-10 أيام بأقل تكلفة ممكنة.

---

## الطريقة المقترحة (منخفضة التكلفة جداً)

### الخطوة 1 — اجمع المادة الخام بأمر واحد

```bash
# آخر commits منذ آخر تحديث في الأرشيف
git log --oneline --format="%ad | %s" --date=short --since="YYYY-MM-DD"

# الملفات الجديدة أو المعدلة
git --no-optional-locks diff --name-only HEAD~20 HEAD

# ملاحظات من الجلسة الحالية إن وجدت
```

### الخطوة 2 — أنشئ ملف الفترة

اطلب من المساعد:

```
أنشئ قيد journal جديد للفترة من [تاريخ] إلى [تاريخ].
اعتمد على:
- هذا الـ git log: [الصق المخرجات]
- ما ناقشناه في هذه الجلسة
- لا تقرأ ملفات إضافية
```

### الخطوة 3 — أضفه للـ INDEX

أضف سطراً في جدول الفترات في `INDEX.md`.

---

## قواعد صارمة لتوفير الرصيد

| ما يُفعل | ما لا يُفعل |
|---------|------------|
| git log فقط كمصدر تقني | قراءة كل الكود من البداية |
| الاعتماد على ذاكرة الجلسة | re-read ملفات كبيرة |
| screenshot واحدة أو اثنتان فقط | screenshots لكل واجهة |
| كتابة جميع الملفات دفعة واحدة (parallel) | كتابة ملف ملف بشكل متسلسل |
| الاكتفاء بالأحداث الهامة | توثيق كل تغيير صغير |

---

## تكرار التحديث المقترح

- **كل 7-10 أيام** من العمل الفعلي
- **عند كل قرار معماري كبير** (مثل: تغيير storage، إضافة نظام جديد)
- **عند حل bug صعب** يستحق التوثيق

---

## هيكل الملف القياسي

```markdown
# الفترة: YYYY-MM-DD → YYYY-MM-DD
**العنوان:** [اسم وصفي للفترة]

## ما تم بناؤه
## المشاكل والأخطاء
## القرارات المعمارية
## الدروس المستفادة
## مصطلحات جديدة
## أفكار محتوى مستقبلي
## لقطات مهمة (اختياري)
```

---

## تنبيه أمني دائم

لا تُدرج أبداً في ملفات الأرشيف:
- عناوين IP حقيقية
- emails شخصية
- مفاتيح API أو secrets
- أسماء مستخدمين حقيقيين
- محتوى terminal prompts حقيقية
- أي بيانات من قاعدة البيانات الحقيقية
