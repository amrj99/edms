## المشاكل والأخطاء

**مشكلة Schema Drift:**
السيناريو الكابوسي: المطور يضيف عمود في الـ database مباشرة (بدون migration file)، يعمل محلياً، ثم في الـ production لا يوجد هذا العمود → crash.

الحل: script يقارن الـ schema الحالي مع آخر migration → يفشل الـ build إذا كان هناك drift.

**مشكلة AI Provider Priority:**
أي الأولوية أعلى: environment variable أم database setting؟
- قرار: env var يتغلب دائماً على database
- السبب: في بيئة production، الـ ops team تريد تغيير الـ provider بدون deploy جديد

**مشكلة Groq Integration:**
Groq يستخدم OpenAI-compatible API لكن مع rate limits مختلفة. التحدي: error handling عند تجاوز الـ rate limit يجب أن يعطي fallback لمزود آخر.

**خطأ في Production Environment Detection:**
```
[startup] Error: Cannot detect production environment
```
السبب: كود كان يعتمد على `NODE_ENV === 'production'` لكن في بعض بيئات Docker القيمة كانت مفقودة.

## القرارات المعمارية

**Credits Threshold للتوجيه:**
- أقل من 50 credit → free tier
- 50 أو أكثر → premium tier
- القيمة 50 اختيرت لتكون "كافية لعدة operations حقيقية" بدون استنزاف سريع

**Database-seeded AI Defaults:**
القيم الافتراضية للـ AI (مزود، نموذج، threshold) تُحفظ في database وليس hardcoded. يسمح بالتغيير دون deploy.

**Schema drift كـ build blocker:**
قرار قوي: إذا كان الـ schema المحلي لا يتطابق مع الـ migrations، الـ build يفشل. لا استثناءات.

## الدروس المستفادة

1. **Schema drift يُكتشف في أسوأ وقت — Production** — الحل: اكتشفه في الـ build
2. **AI providers تفشل — صمم للـ fallback من البداية** — لا تعتمد على مزود واحد
3. **Environment variables > database settings للـ ops-critical config** — flexibility عند الطوارئ
4. **Privacy by design** — إضافة privacy mode لاحقاً أصعب من بنائه في البداية
5. **Docker compose needs explicit env vars** — لا تفترض أن الـ container يرث الـ env من الـ host
