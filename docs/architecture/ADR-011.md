# ADR-011 — orgScopedWhere: حد العزل الأساسي بين المستأجرين

| الحقل | القيمة |
|-------|--------|
| **الحالة** | `ACCEPTED` |
| **القرار اتُّخذ في** | Phase 5 Design Review — 2026-07-05 |
| **الجمهور** | أي مطور يُعدِّل routes أو `lib/org-scope.ts` |
| **الملف المرجعي** | `artifacts/api-server/src/lib/org-scope.ts` |

---

## السياق

ArcScale SaaS تُشغِّل multiple tenants على PostgreSQL instance واحدة. الـ tenant isolation مطبَّق في application layer عبر SQL WHERE clauses. لا يوجد PostgreSQL Row-Level Security (RLS) — انظر ADR-06.

مع إضافة Party Model (Phase 5)، نشأت الحاجة لأول مرة لوصول Cross-Organization داخل نفس الـ instance: منظمة خارجية (Org B) تحتاج الوصول لبيانات مشروع تملكه Org A.

هذا يُثير خطر تخفيف حد العزل الموجود — إما بتعديل `orgScopedWhere` مباشرة أو بتمرير `organizationId` الأجنبي إليها.

---

## القرار

**`orgScopedWhere()` في `lib/org-scope.ts` هو حد العزل الوحيد بين المستأجرين على مستوى SQL.**

### ما تفعله الدالة

```typescript
// system_owner → WHERE id = ?
// all others   → WHERE id = ? AND organization_id = caller.organizationId
```

هذا الـ invariant لا يُكسَر. الدالة لا تقبل `organizationId` اختيارياً، ولا تقبل قائمة من المنظمات، ولا تقبل OR clause.

---

## القواعد الإلزامية

### 1. لا تُعدَّل `orgScopedWhere` لأي سبب يتعلق بالتعاون بين المنظمات

الأشكال المحظورة:

```typescript
// ❌ محظور — جعل org اختيارية
orgScopedWhere(caller, table.id, id, table.orgId, optionalPartyOrgId?)

// ❌ محظور — OR clause
WHERE id = ? AND (org_id = caller.orgId OR project_id IN (...))

// ❌ محظور — تمرير org أجنبية
orgScopedWhere({ ...caller, organizationId: projectOwnerOrgId }, ...)
```

### 2. أي وصول Cross-Organization يسلك هذا المسار حصراً

```
canAccessProjectAsParty(caller.organizationId, projectId)   ← التحقق
           ↓
  WHERE project_id = ?                                       ← بيانات المشروع
           ↓
  assertPartyCeiling(mode, partyRole, action)                ← السقف
```

المسار موجود في `lib/party-access.ts`. لا يتقاطع مع `lib/org-scope.ts`.

### 3. المسار الداخلي (intra-org) لا يتغير

```typescript
// الشكل 1 — scoped mutation
const filter = orgScopedWhere(caller, table.id, id, table.orgId);
await db.update(table).set({...}).where(filter);

// الشكل 2 — check-after-fetch
const [row] = await db.select().where(eq(table.id, id)).limit(1);
if (!assertOrgMatch(req, res, row.organizationId)) return;
```

---

## الاستدلال

### لماذا لا نُعدِّل `orgScopedWhere`؟

`orgScopedWhere` مُستخدم في عشرات الـ routes. أي تغيير في توقيعها ينتشر أثره فورياً عبر كل routes الـ API. الـ party access هو استثناء محدود النطاق (مشروع واحد، سقف محدد) — لا يستحق تغيير الـ primitive الأساسي.

### لماذا `WHERE project_id = ?` للـ party queries؟

بيانات المشروع (documents, transmittals...) تحمل `organizationId = projectOwnerOrgId`. استخدام `orgScopedWhere` بـ `caller.organizationId` (الذي هو OrgB) يُرجع صفر نتائج — لأن البيانات لـ OrgA.

بديل آمن: بعد التحقق من صلاحية الـ party access، نستخدم `WHERE project_id = ?` — يحصر البيانات لهذا المشروع فقط، لا لكل OrgA.

---

## البدائل المرفوضة

| البديل | سبب الرفض |
|--------|-----------|
| تعديل `orgScopedWhere` لقبول `orgId` اختياري | undefined behavior — system_owner bypass ينطبق بشكل غير متوقع، تعقيد كل استدعاء موجود |
| إضافة OR clause في الـ WHERE | يُعقِّد كل query، يصعب الاستدلال، يخلط بين مسارَي الأمان |
| استخدام `projectOwnerOrgId` كـ caller.organizationId للـ party query | يكشف كل بيانات OrgA (ليس فقط المشروع المحدد) |
| RLS في PostgreSQL بديلاً | مُقيَّم في ADR-06 كـ ACCEPTED_RISK — لا يُعفي من application-layer isolation |

---

## النتائج

- intra-org queries: `orgScopedWhere` دائماً — لا استثناء
- party queries: `WHERE project_id = ?` + ceiling enforcement — لا `orgScopedWhere`
- أي PR يُعدِّل `lib/org-scope.ts` يتطلب تبريراً معمارياً صريحاً في وصف الـ PR
- اختبارات PM-16 و PM-17 في `test/party-access.test.ts` تُثبت صحة الفصل (intra-org لا يزال يعمل بعد إضافة party branch)
