## 📋 EDMS — Register System Development

### Context
We need to build three main registers in the system:
- **Master Document Register** — tracks all project documents
- **Drawings Register** — tracks drawings by type (Design / Shop / IFC / As-Built)
- **Transmittal Register** — tracks all outgoing and incoming transmittals

Registers are **not** separate database tables — they are filtered queries/views on existing tables with smart filtering by party, direction, discipline, and status.

### 2️⃣ Automatic Document Status Update Logic

When an **incoming** transmittal is updated and its `approvalStatus` changes, the system must automatically update the status of all linked documents. Add this logic inside the transmittal update endpoint:

```typescript
// After saving the transmittal update, run this logic:

if (
  transmittal.direction === "incoming" &&
  updatedData.approvalStatus === "approved"
) {
  const items = await db
    .select()
    .from(transmittalItemsTable)
    .where(eq(transmittalItemsTable.transmittalId, transmittalId));

  for (const item of items) {
    let newDocStatus: DocumentStatus | null = null;

    switch (item.reviewCode) {
      case "A": newDocStatus = "approved"; break;
      case "B": newDocStatus = "approved_with_comments"; break;
      case "C": newDocStatus = "for_revision"; break;
      case "D": newDocStatus = "rejected"; break;
      default:  break; // UR or null = no change
    }

    if (newDocStatus) {
      // Safety check: do not downgrade a document that is already in a higher state
      const currentDoc = await db
        .select()
        .from(documentsTable)
        .where(eq(documentsTable.id, item.documentId))
        .then(r => r[0]);

      const noDowngrade = ["issued", "superseded", "void"];
      if (currentDoc && noDowngrade.includes(currentDoc.status)) continue;

      // 1. Update document status
      await db
        .update(documentsTable)
        .set({ status: newDocStatus, updatedAt: new Date() })
        .where(eq(documentsTable.id, item.documentId));

      // 2. Log to document_revisions for full traceability
      await db.insert(documentRevisionsTable).values({
        documentId: item.documentId,
        organizationId: transmittal.organizationId,
        revision: item.revision ?? "—",
        status: newDocStatus,
        comment: `Auto-updated via Transmittal ${transmittal.transmittalNumber} — Review Code ${item.reviewCode}`,
        createdById: transmittal.createdById,
        reviewDecision: item.reviewCode,
        reviewerName: transmittal.toExternal ?? undefined,
      });
    }
  }
}
```

### 5️⃣ Important Implementation Notes

**Multi-tenancy:**
- All new tables must include `organization_id` for proper multi-tenant row-level isolation
- Verify that `transmittals` table has `organization_id` — add it if missing

**Column config inheritance:**
- Project-level config overrides org-level config
- Org-level config overrides system defaults
- Always fall back gracefully through this chain

**Auto status update rules:**
- Never downgrade a document already in `issued`, `superseded`, or `void` status
- Always write a `document_revisions` entry when auto-updating, referencing the transmittal number
- Only trigger auto-update when `direction = "incoming"` — outgoing transmittals do not update document status

**Register query performance:**
- Add database indexes on `documents.party_type`, `documents.drawing_type`, `transmittals.direction`, and `transmittals.party_type`
- The `transmittal-history` endpoint should join `transmittals`, `transmittal_items`, and `document_revisions` in a single query

**party_type values (use consistently across all tables):**
```
"owner"            → Client / Employer
"consultant"       → Engineer / Consultant
"main_contractor"  → Main Contractor
"subcontractor"    → Subcontractor
"authority"        → Government Authority (used mainly for NOC)
```
