## 📋 المطلوب: تطوير نظام الريجيسترات في EDMS

### السياق
نحتاج بناء ثلاثة ريجيسترات رئيسية في النظام:
- **Master Document Register** — سجل كل المستندات
- **Drawings Register** — سجل الرسومات (Design / Shop / IFC / As-Built)
- **Transmittal Register** — سجل عمليات الإرسال والاستقبال

الريجيسترات ليست جداول منفصلة — هي queries/views على الجداول الموجودة مع فلترة ذكية.

**أ) في `transmittals.ts` — أضف حقلين:**

**ج) في `transmittal_items` — أضف ثلاثة حقول:**

**د) في `metadata.ts` — عدّل enum `metadata_applies_to` ليشمل:**

**هـ) جدول جديد كامل — `register_column_config`:**
```typescript
import { pgTable, serial, text, timestamp, integer, boolean } from "drizzle-orm/pg-core";
import { organizationsTable } from "./organizations";
import { projectsTable } from "./projects";

export const registerColumnConfigTable = pgTable("register_column_config", {
  id: serial("id").primaryKey(),
  organizationId: integer("organization_id")
    .references(() => organizationsTable.id)
    .notNull(),
  projectId: integer("project_id")
    .references(() => projectsTable.id),
  // null = إعداد افتراضي على مستوى المؤسسة
  // إذا وُجد projectId فهو يتجاوز الإعداد الافتراضي

  registerType: text("register_type").notNull(),
  // القيم: "transmittal" | "document" | "drawing" | "ncr" | "itr" | "noc" | "correspondence"

  columnKey: text("column_key").notNull(),
  // أمثلة: "discipline" | "drawing_type" | "review_code" | "party_type" | "area_location" | "itp_ref"

  isVisible: boolean("is_visible").notNull().default(true),
  displayOrder: integer("display_order").notNull().default(0),
  columnLabel: text("column_label"),
  // اسم مخصص للعمود — إذا كان null يستخدم النظام الاسم الافتراضي

  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

### 2️⃣ منطق التحديث التلقائي لحالة المستند

عند تحديث ترانسميتال وارد (incoming) وتغيير حالته، يجب تنفيذ هذا المنطق تلقائياً في الـ endpoint المسؤول عن تحديث الترانسميتال:

```typescript
// بعد حفظ تحديث الترانسميتال، نفّذ هذا المنطق:

if (transmittal.direction === "incoming" && updatedData.approvalStatus === "approved") {
  const items = await db
    .select()
    .from(transmittalItemsTable)
    .where(eq(transmittalItemsTable.transmittalId, transmittalId));

  for (const item of items) {
    let newDocStatus: DocumentStatus | null = null;

    switch (item.reviewCode) {
      case "A":  newDocStatus = "approved"; break;
      case "B":  newDocStatus = "approved_with_comments"; break;
      case "C":  newDocStatus = "for_revision"; break;
      case "D":  newDocStatus = "rejected"; break;
      default:   break; // UR أو null = لا تغيير
    }

    if (newDocStatus) {
      // 1. حدّث حالة المستند
      await db
        .update(documentsTable)
        .set({ status: newDocStatus, updatedAt: new Date() })
        .where(eq(documentsTable.id, item.documentId));

      // 2. سجّل في document_revisions للتتبع
      await db.insert(documentRevisionsTable).values({
        documentId: item.documentId,
        organizationId: transmittal.organizationId, // أضف organizationId للترانسميتال إن لم يكن موجوداً
        revision: item.revision ?? "—",
        status: newDocStatus,
        comment: `Auto-updated via Transmittal ${transmittal.transmittalNumber} — Code ${item.reviewCode}`,
        createdById: transmittal.createdById,
        reviewDecision: item.reviewCode,
        reviewerName: transmittal.toExternal ?? undefined,
      });
    }
  }
}
```

### 3️⃣ API Endpoints الجديدة المطلوبة

```
// ─── ريجيسترات القراءة ───────────────────────────────────────────
GET /api/projects/:projectId/registers/documents
GET /api/projects/:projectId/registers/drawings
GET /api/projects/:projectId/registers/transmittals

// Query params مشتركة لكل الريجيسترات:
?partyType=consultant         // فلتر حسب الجهة
&direction=incoming           // فلتر حسب الاتجاه (transmittal فقط)
&discipline=Civil             // فلتر حسب التخصص
&drawingType=shop             // فلتر حسب نوع الرسمة (drawings فقط)
&status=approved              // فلتر حسب الحالة
&reviewCode=A                 // فلتر حسب كود المراجعة
&search=SD-101                // بحث نصي في الرقم والعنوان
&page=1&limit=50              // pagination

// ─── تاريخ مستند معين ────────────────────────────────────────────
GET /api/projects/:projectId/documents/:documentId/transmittal-history
// يُرجع: كل الترانسميتالز التي مر بها المستند مرتبة زمنياً
// مع: direction، partyType، reviewCode، تواريخ

// ─── إعدادات أعمدة الريجيستر ─────────────────────────────────────
GET    /api/projects/:projectId/register-config/:registerType
PUT    /api/projects/:projectId/register-config/:registerType
// يسمح بإظهار/إخفاء الأعمدة وترتيبها لكل مشروع
// إذا لم يوجد إعداد للمشروع، يرجع إعداد المؤسسة الافتراضي
```

### 5️⃣ ملاحظات مهمة للتنفيذ

- جميع الجداول الجديدة يجب أن تحتوي `organization_id` لضمان **multi-tenant isolation**
- الترانسميتال يحتاج إضافة `organization_id` إن لم يكن موجوداً — تحقق من ذلك
- إعداد الأعمدة يتوارث: إعداد المشروع يتجاوز إعداد المؤسسة، وإعداد المؤسسة يتجاوز الافتراضي
- عند تحديث حالة المستند تلقائياً، لا تتجاوز الحالة إذا كانت أعلى — مثلاً مستند `issued` لا يتراجع إلى `approved`
- الـ `transmittal-history` endpoint يجب أن يكون مرتباً تصاعدياً حسب `sent_at`
